# Application Dependency & Capability Spec – Conversation Summary

## Purpose of This Document
This document summarizes the design discussion around a proposed specification for describing **application integration dependencies** and how platforms fulfill them. The goal is to standardize the handshake between:

- **What an application requires to run successfully**
- **What a platform offers to satisfy those requirements**
- **How those requirements are bound to concrete infrastructure**

The conversation focused on identifying the **core data sets**, clarifying their semantic roles, simplifying the model, and mapping the design to existing ecosystem projects.

---

# Core Architectural Pipeline

The system is conceptually modeled as a pipeline:

ObservedBehavior → Condition → CapabilityCatalog → BindingSet

| Layer | Meaning |
|------|------|
| ObservedBehavior | Evidence of real runtime behavior |
| Condition | Portable description of required capabilities |
| CapabilityCatalog | Platform-side catalog of offered capabilities |
| BindingSet | Concrete mapping between requirements and implementations |

This pipeline separates **facts**, **requirements**, **platform offerings**, and **realization decisions**.

---

# Core Data Sets (Primary Spec Objects)

After discussion, the spec focuses on **four primary data sets**.

## 1. ObservedBehavior
**Purpose:** Evidence describing what the application actually did during execution.

ObservedBehavior records runtime facts detected by an observation engine (for example eBPF or runtime tracing). These facts become the foundation for identifying integration dependencies.

Examples:

- outbound TCP connection to port 5432
- HTTP requests to `/api/users`
- Redis protocol usage
- environment variable reads

ObservedBehavior is the most stable and concrete data set because it represents **objective runtime evidence**.

Typical producers:

- Kubescape
- eBPF profilers
- runtime instrumentation tools

Role in spec:

ObservedBehavior provides the **input evidence** from which Conditions can be generated.

---

## 2. Condition
**Purpose:** Portable declaration of an application's dependency.

Conditions translate runtime evidence into **environment-agnostic requirements**.

Example transformation:

ObservedBehavior:

```
tcp_connect: 10.0.14.2:5432
protocol: postgres
```

Condition:

```
kind: datastore.relational
interface:
  protocol: postgres
criticality: runtime-required
```

A Condition answers:

> What capability must exist for the application to function?

Key characteristics:

- environment-agnostic
- platform-agnostic
- capability-focused

Conditions represent the **portable dependency contract** for an application.

---

## 3. CapabilityCatalog
**Purpose:** Platform-side declaration of capabilities offered to applications.

This catalog defines what a platform team provides as reusable services.

Example concept:

```
capabilityId: relational-db
kind: datastore.relational
requestSchema:
  properties:
    engine
    size
bindingContract:
  outputs:
    endpoint
    username
    password
```

The catalog may be **environment-scoped**, meaning dev and prod environments can provide the same capability but with different implementations.

Example:

| Environment | Implementation |
|-------------|---------------|
| dev | PostgreSQL Deployment |
| prod | AWS RDS cluster |

The catalog therefore represents the **supply side of platform capabilities**.

Promoting the CapabilityCatalog became a key design decision in the discussion.

---

## 4. BindingSet
**Purpose:** Concrete mapping between Conditions and real platform implementations.

BindingSets are generated by matching application requirements to the capabilities offered by a platform.

Example:

```
conditionRef: relational-db
implementation:
  provider: aws
  service: rds-postgres
connection:
  secretRef: checkout-db-conn
```

BindingSets answer:

> How will this requirement be satisfied in this environment?

Characteristics:

- usually generated automatically
- may be refined by platform engineers
- results in deployable infrastructure or services

---

# Reduction of EnvironmentProjection

Initially, the architecture included a resource called **EnvironmentProjection**.

Its purpose was to describe:

- target environment
- provider
- region
- policies
- preferences

Example concept:

```
environment: aws
region: us-east-1
managed-services-preferred: true
```

## Decision from Discussion

EnvironmentProjection was determined to be **loosely defined and redundant**.

Instead of exposing platform intent in the spec:

- platform teams should encode environment-specific behavior **inside the CapabilityCatalog**
- dev/prod differences become part of catalog offerings

Example:

| Capability | Dev Implementation | Prod Implementation |
|-----------|-------------------|--------------------|
| relational-db | Postgres Deployment | AWS RDS |
| cache | Redis Pod | Elasticache |

Therefore:

**EnvironmentProjection is no longer required as a core spec object.**

Its responsibilities move into the platform's capability definitions.

---

# Remaining Data Sets in the Ecosystem Model

Other objects described in the architecture remain valuable but are considered **adjacent or extension resources** rather than core spec primitives.

| Resource | Category |
|---------|----------|
| ObservationEngine | evidence metadata |
| ObservationSet | evidence grouping |
| WorkloadCatalog | workload identity mapping |
| ConditionSet | container object |
| CapabilityBinding | internal platform resolution |
| ProviderResolver | platform-specific mapping |
| Assertion | validation rules |
| AssertionResult | validation results |
| ResourceBundle | generated infrastructure |
| NetworkPolicyBundle | security artifacts |
| MockBundle | testing artifacts |
| SchemaBundle | contract definitions |

These can be treated as **profiles or extensions** rather than normative spec objects.

---

# Relationship Between Core Objects

The simplified architecture becomes:

```
ObservedBehavior
      ↓
Condition
      ↓
CapabilityCatalog
      ↓
BindingSet
```

Responsibilities:

| Layer | Owner |
|-----|-----|
| ObservedBehavior | observation tooling |
| Condition | application developers / automation |
| CapabilityCatalog | platform team |
| BindingSet | automation / platform control plane |

# Alignment With Existing Projects

The discussion evaluated several projects to determine how they align with the core data sets.

---

# Score

Entity: Score

Score is a **workload specification** that lets applications declare required resources.

### Mapping

| Spec Dataset | Score Equivalent |
|---------------|-----------------|
| Condition | Score `resources` |
| CapabilityCatalog | not standardized |
| BindingSet | produced by Score implementations |

Example:

```
resources:
  db:
    type: postgres
```

Score is the **closest existing model for Conditions**.

---

# Radius

Entity: Radius

Radius models applications using resource types and deployment recipes.

### Mapping

| Spec Dataset | Radius Equivalent |
|--------------|------------------|
| Condition | Resource Type |
| CapabilityCatalog | Resource Types + Environment |
| BindingSet | Recipes |

Radius Environments select which recipes apply to each resource type, enabling environment-specific bindings.

Radius is a strong candidate for **BindingSet generation**.

---

# Crossplane

Entity: Crossplane

Crossplane provides platform APIs using composite resources.

### Mapping

| Spec Dataset | Crossplane Equivalent |
|--------------|----------------------|
| Condition | composite resource request |
| CapabilityCatalog | XRD (Composite Resource Definition) |
| BindingSet | Composition |

Example:

XRD defines capability:

```
kind: RelationalDatabase
```

Composition defines implementation:

```
AWS RDS
```

or

```
Postgres Operator
```

Crossplane is a strong example of **CapabilityCatalog + BindingSet**.

---

# Kratix

Entity: Kratix

Kratix exposes platform services through **Promises**.

### Mapping

| Spec Dataset | Kratix Equivalent |
|--------------|------------------|
| Condition | Promise request |
| CapabilityCatalog | Promise |
| BindingSet | Promise workflow |

Kratix is one of the clearest **platform capability catalog** implementations.

---

# kro (Kube Resource Orchestrator)

Entity: kro

kro defines platform abstractions using ResourceGraphDefinitions.

### Mapping

| Spec Dataset | kro Equivalent |
|--------------|---------------|
| Condition | RGD instance |
| CapabilityCatalog | ResourceGraphDefinition |
| BindingSet | generated Kubernetes resources |

kro provides another example of Kubernetes-native capability catalogs.

---

# Open Service Broker API

Entity: Open Service Broker API

The Open Service Broker API defines a **service catalog**.

### Mapping

| Spec Dataset | OSB Equivalent |
|--------------|---------------|
| Condition | service request |
| CapabilityCatalog | service catalog |
| BindingSet | service binding |

OSB is the closest thing to a **standardized capability catalog** today.

---

# Dapr

Entity: Dapr

Dapr provides runtime abstractions like:

- state store
- pub/sub
- bindings

### Mapping

| Spec Dataset | Dapr Equivalent |
|--------------|----------------|
| Condition | component type |
| CapabilityCatalog | component ecosystem |
| BindingSet | component configuration |

Dapr focuses more on **runtime abstraction** than catalog modeling.

---

# Open Application Model (OAM)

Entity: Open Application Model

OAM separates application configuration from operator concerns.

### Mapping

| Spec Dataset | OAM Equivalent |
|--------------|---------------|
| Condition | component spec |
| CapabilityCatalog | workload types |
| BindingSet | traits |

OAM is more conceptual but supports similar separation of responsibilities.

# Final Design Direction

## Core Spec Responsibilities

The proposed specification focuses on standardizing:

1. **ObservedBehavior** – runtime evidence  
2. **Condition** – portable application requirements  
3. **CapabilityCatalog** – platform capability offerings  
4. **BindingSet** – resolution between requirements and offerings  

## Removed / Reduced Elements

EnvironmentProjection is no longer considered a required core resource.

Its semantics move into **environment-scoped CapabilityCatalog entries**.

## Ecosystem Integration

Existing projects can adopt different parts of the spec:

| Layer | Likely Integrations |
|------|---------------------|
| ObservedBehavior | Kubescape / eBPF tools |
| Condition | Score |
| CapabilityCatalog | Crossplane / Kratix / kro / OSB |
| BindingSet | Radius / Crossplane / Kratix |

This creates a potential ecosystem architecture:

```
Observation Engine
        ↓
     Condition
        ↓
CapabilityCatalog (platform)
        ↓
     BindingSet
        ↓
Infrastructure + Policies + Mocks
```

---

# Conclusion

The conversation produced several important outcomes:

- clarified the semantic meaning of the main data sets
- simplified the architecture
- reduced reliance on EnvironmentProjection
- elevated the importance of CapabilityCatalog
- mapped the model to existing open source projects

The resulting architecture emphasizes a **clear separation of responsibilities**:

| Concern | Responsibility |
|-------|---------------|
| Evidence | runtime observation |
| Meaning | application requirements |
| Supply | platform capabilities |
| Realization | automated binding |

This separation provides a strong foundation for a portable specification describing **how applications declare dependencies and how platforms satisfy them**.
